# 🛸 无人机航线规划系统 - 算路因子配置参考指南

## 1. 核心评估模型 (Evaluation Model)
本系统采用基于网格的 A* (A-Star) 启发式搜索算法。在 3D 空间中，算法通过计算每个网格节点的**“总代价 (Total Cost)”**来寻找最优航线。

总代价的计算融合了物理距离与多维度的环境风险因子，其核心数学模型如下：

$$TotalCost = DistanceCost + \sum_{i=1}^{n} (FactorWeight_{i} \times FactorCost_{i})$$

* **$FactorWeight$ (权重)**：系统对该类风险的重视程度。权重越高，算法越倾向于规避该风险。可由 API 动态覆盖。
* **$FactorCost$ (代价)**：节点在该因子下的客观得分（范围通常为 `0.0` ~ `1.0`）。如：14m/s的风速会导致代价飙升至 1.0。

## 2. 算路模式预设 (Route Mode)
系统后端 (`GridEvaluator`) 提供了三种宏观调优模式，在 API 请求体中通过 `mode` 字段声明：

| 模式名称 | `mode` 传参 | 后端定义 | 核心策略 |
| :--- | :--- | :--- | :--- |
| **最短路径** | `"shortest"` | `SHORTEST` | **效率极化**。大幅放大 `distance` 权重，只要没有绝对禁飞区，一律走直线。 |
| **风险最低** | `"safest"` | `SAFEST` | **安全极化**。大幅放大 `risk_area` 和气象权重，宁愿增加 50% 航程也要绕开敏感区。 |
| **综合最优** | `"balanced"` | `BALANCED` | **科学平权**。将效率、安全、合规按黄金比例配平，寻找整体成本最低的航线。 |

## 3. 核心因子数据字典 (Factor Dictionary)

以下列出了系统中支持动态配置的所有算路因子。**注：表格中的“默认权重”为 C++ 后端预设值，在 API 中传入新 `weight` 将覆盖这些预设值。**

### 3.1 气象环境因子 (`wdh_11`)
控制飞机对恶劣天气的规避倾向。

| 因子字段 | 说明 | 默认权重 | 代价阶梯 (Cost) |
| :--- | :--- | :--- | :--- |
| `windSpeed` | 实时/预测风速 | `0.0203` | `[0,5]m/s` $\rightarrow$ 0.0 <br> `(10,13.8]m/s` $\rightarrow$ 0.8 <br> `>13.8m/s` $\rightarrow$ 1.0 (高危) |
| `rainPcpn` | 降水量 | `0.0081` | 无雨 $\rightarrow$ 0.0 <br> `>8mm` $\rightarrow$ 1.0 (大暴雨极力规避) |
| `visibility` | 能见度 | `0.0054` | `>5000m` $\rightarrow$ 0.0 <br> `<2000m` $\rightarrow$ 1.0 (易发撞击危险) |
| `temperature`| 温度 | `0.0045` | `0~35°C` $\rightarrow$ 0.0 <br> `<0°C` $\rightarrow$ 0.6 (电池衰减惩罚) |

### 3.2 信号覆盖因子 (`cns_14`)
控制飞机在断网/失联区域的飞行策略。

| 因子字段 | 说明 | 默认权重 | 代价阶梯 (Cost) |
| :--- | :--- | :--- | :--- |
| `communication`| 通信链路 (4G/5G) | `0.0338` | 有信号 $\rightarrow$ 0.1，无信号 $\rightarrow$ 0.2 |
| `navigation` | 卫星导航 (GNSS) | `0.0263` | 有导航 $\rightarrow$ 0.1，无导航 $\rightarrow$ 0.2 |
| `surveillance` | 监视信号 (ADS-B) | `0.0150` | 有监视 $\rightarrow$ 0.1，无监视 $\rightarrow$ 0.2 |

### 3.3 合规与地面风险因子 (`risk_14`, `privacy_14`)
控制飞行的社会影响与安全性。**该类属高敏因子，权重占比极大。**

| 因子字段 | 说明 | 默认权重 | 代价阶梯 (Cost) |
| :--- | :--- | :--- | :--- |
| `risk_area` | 地面风险 (如人群) | `0.1000` | 低风险 $\rightarrow$ 0.3，高风险 $\rightarrow$ 1.0 |
| `residential_area`| 居住/隐私区 | `0.1000` | 非居住区 $\rightarrow$ 0.0，居住区 $\rightarrow$ 0.5 |

## 6. Redis 动态数据缓存字典 (Redis Cache Mapping)

在 A* 算法执行期间，系统需要对每个经过的三维网格（Grid）进行代价计算。为了保证寻路性能，所有的实时环境、信号和风险数据均通过 **Redis** 进行高速读取。

### 6.1 Redis Key 命名规范
系统采用标准的分层 Key 命名结构。通过将配置参数中的“大类”与“子类”组合作为前缀，拼接网格的唯一空间编码（如 Geohash、H3 索引或平面网格 XY 坐标），实现 `O(1)` 的高频查询：

> **标准格式**： `{模块前缀}:{子因子字段}:{网格空间编码}`
> **示例**： `wdh_11:windSpeed:wtw3sjq`

### 6.2 核心前缀匹配清单

后端引擎在评估某一网格点时，需向 Redis 批量或管道（Pipeline）请求以下前缀的数据：

#### 🌥️ 气象环境类数据 (`wdh_11:`)
*通常具有时效性（TTL），定期通过气象预报服务刷新。*
| JSON 因子分类 | 对应的 Redis 查询前缀 | Key 示例 | 预期返回数据类型 |
| :--- | :--- | :--- | :--- |
| `windSpeed` | `wdh_11:windSpeed:*` | `wdh_11:windSpeed:1024_2048` | `Float` (如 `6.5` m/s) |
| `rainPcpn` | `wdh_11:rainPcpn:*` | `wdh_11:rainPcpn:1024_2048` | `Float` (如 `1.2` mm) |
| `visibility` | `wdh_11:visibility:*` | `wdh_11:visibility:1024_2048` | `Float` (如 `3500.0` m) |
| `temperature` | `wdh_11:temperature:*` | `wdh_11:temperature:1024_2048` | `Float` (如 `32.5` °C) |

#### 📡 基础设施与信号数据 (`cns_14:`)
*相对静态，结合运营商基站数据与空管数据更新。*
| JSON 因子分类 | 对应的 Redis 查询前缀 | Key 示例 | 预期返回数据类型 |
| :--- | :--- | :--- | :--- |
| `communication` | `cns_14:communication:*` | `cns_14:communication:1024_2048` | `Int` (如 `85` 信号强度) |
| `navigation` | `cns_14:navigation:*` | `cns_14:navigation:1024_2048` | `Int` (如 `12` 卫星数) |
| `surveillance` | `cns_14:surveillance:*` | `cns_14:surveillance:1024_2048` | `Int` (如 `100` 覆盖率) |

#### ⚠️ 风险与合规数据 (`risk_14:` / `privacy_14:`)
*结合城市 GIS 数据网格化生成。*
| JSON 因子分类 | 对应的 Redis 查询前缀 | Key 示例 | 预期返回数据类型 |
| :--- | :--- | :--- | :--- |
| `risk_area` | `risk_14:risk_area:*` | `risk_14:risk_area:1024_2048` | `Int` (如 `1/2/3` 风险等级) |
| `residential_area`| `privacy_14:residential_area:*` | `privacy_14:residential_area:1024` | `Int` (如 `0/1` 是否居住区) |

#### 🚫 物理障碍与地形数据
*结合 OSGB 切片提取的基础约束，直接阻断通行。*
| JSON 因子分类 | 对应的 Redis 查询前缀 | 物理含义 | 预期返回数据类型 |
| :--- | :--- | :--- | :--- |
| `za_14` | `za_14:*` | 障碍物 (Obstacles) | `Bool` / `1` (命中即不可飞) |
| `dz_14` | `dz_14:*` | 禁飞区/地质区 | `Bool` / `1` (命中即绕飞) |

### 6.3 缓存未命中（Cache Miss）与兜底逻辑
为了保证算法的鲁棒性，当引擎拼接好 Redis Key 尝试获取网格数据，但发生 `Key NOT FOUND` 或 Redis 服务抖动时，**绝对不能中断规划或默认取 0**。

**兜底策略 (Fallback)**：
1. 引擎必须立即读取本次 API 请求 `condition` JSON 中传入的 **`defaultValue`**（默认值）。
2. 将 `defaultValue` 带入 `range` 区间匹配逻辑中，计算出该缺失因子的默认 Cost。
3. *例如：当 `cns_14:communication:xxx` 查不到数据时，将其默认视为处于未知区域，直接取 `defaultValue: 0.2` 作为高昂的通信惩罚代价，促使算法偏向选择有明确信号数据的网格。*
